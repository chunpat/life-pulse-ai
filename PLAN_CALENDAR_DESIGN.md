# LifePulse Plan 与 App 内日历设计

## 1. 目标

在现有一句话记录基础上，新增一类面向“行程安排 / 待办提醒 / 约会事件”的 Plan 能力，并提供：

1. App 内可视化日历
2. 一句话创建计划
3. 计划的查看、编辑、完成、取消
4. 与 iOS 提醒事项、日历的联动

这套能力与现有 Goal 不同。

- Goal: 连续习惯与挑战，例如 7 天早睡、21 天运动
- Plan: 带时间属性的任务或事件，例如 明天下午三点开会、周五提醒交房租

## 2. 产品定位

Plan 是一个独立模块，不应直接复用 Goal 数据结构。

原因：

1. Goal 的核心是连续天数、打卡进度、奖励积分
2. Plan 的核心是时间、提醒、日历视图、同步状态
3. 如果把 Plan 强塞进 Goal，后续编辑时间、重复规则、外部同步会让模型越来越脏

## 3. 信息架构

建议在现有底部导航中新增一个一级入口：

1. 时间线
2. 计划
3. 新建
4. 分析

财务仍保留在顶部入口，不需要塞进底部主导航。

Plan 一级页内部再分成三个切换：

1. 今日
2. 日历
3. 清单

默认进入“今日”。

## 4. App 内日历为什么必须做

即使已经和 iOS 原生日历、提醒事项联动，App 内日历仍然必须存在。

原因：

1. 用户需要一个统一视图，能同时看到应用内计划和同步到系统的事项
2. iOS 原生数据不一定始终可读，权限、账号、网络都可能影响联动
3. 你需要在 App 内承载 AI 解析、编辑确认、冲突处理、同步状态展示
4. 日后 Android 也会需要同一套产品层，不应该完全依赖 iOS 原生界面

结论：

- 原生日历和提醒事项是同步目标
- App 内日历才是主工作台

## 5. 核心页面设计

### 5.1 今日页

这是 Plan 模块的默认首页，强调今天应该做什么。

页面结构建议：

1. 顶部日期条
   显示今天日期、星期、快速切换到昨天 / 明天
2. AI 一句话输入框
   例如：明天下午 3 点和产品开会
   例如：周五提醒我交房租
3. 今日时间轴
   按时间顺序展示今天的计划事件
4. 今日待办区
   展示今天到期但未完成的提醒事项
5. 同步状态条
   展示是否已连接 iOS 提醒事项 / 日历

适合承接高频动作：

1. 快速创建
2. 快速完成
3. 快速改时间
4. 快速同步

### 5.2 日历页

这是 App 内日历主视图，负责“看全局安排”。

建议提供三种视图：

1. 月视图
   用于看哪天有安排、密度如何
2. 周视图
   用于看每天时间分布
3. 日视图
   用于查看当天事件细节和空闲时间段

建议优先级：

1. 先做月视图 + 当日详情抽屉
2. 再做周视图
3. 最后做完整日程时间轴

月视图单元格建议展示：

1. 日期
2. 计划数量角标
3. 颜色短条
   蓝色表示日历事件
   琥珀色表示提醒事项
   绿色表示已完成
   红色表示已过期未完成

点击某一天后，底部弹出当日计划抽屉。

### 5.3 清单页

这是偏 GTD / 提醒事项的视图，不按时间栅格展示。

建议分组：

1. 今天
2. 即将到来
3. 已逾期
4. 已完成

适合处理：

1. 批量勾选完成
2. 修改提醒时间
3. 搜索标题
4. 查看同步来源

## 6. 计划类型设计

Plan 至少分两类：

1. reminder
   更像任务、待办、提醒事项
2. event
   更像会议、约会、出行、课程

推荐规则：

1. 有明确开始时间和结束时间，优先判定为 event
2. 只有截止时间或提醒时间，优先判定为 reminder
3. 没有识别出时间时，允许保存为 inbox 状态，等用户补全

## 7. 一句话录入流程

复用现有 Logger 的自然语言输入体验，但输出到 Plan。

输入示例：

1. 明天下午三点和产品开需求会
2. 周五提醒我交房租
3. 今晚八点去健身
4. 下周一上午十点体检，提前一天提醒

建议流程：

1. 用户输入一句话
2. AI 做意图识别
   是生活日志、计划、还是财务
3. 若识别为计划，解析结构化字段
4. 弹出确认卡片
5. 用户一键保存或微调
6. 根据类型决定是否同步到 iOS 提醒事项或日历

确认卡片字段建议：

1. 标题
2. 类型
3. 日期
4. 开始时间
5. 结束时间
6. 提醒时间
7. 同步目标
8. 备注

## 8. 数据模型建议

### 8.1 Plan

建议新增 Plan 表。

核心字段：

1. id
2. userId
3. title
4. notes
5. planType
   reminder | event
6. status
   pending | completed | cancelled
7. startAt
8. endAt
9. dueAt
10. isAllDay
11. timezone
12. reminderAt
13. recurrenceRule
14. source
   manual | ai | imported
15. syncTarget
   none | ios-reminder | ios-calendar
16. syncState
   local-only | synced | pending-sync | conflict | permission-denied | failed
17. externalId
18. externalContainerId
19. metadata

### 8.2 PlanOccurrence

如果后面要支持重复计划，建议预留 occurrence 设计。

可以先不建表，但模型上要预留演进空间。

### 8.3 PlanChangeLog

用于记录冲突和同步过程中的关键变更。

字段建议：

1. id
2. planId
3. action
4. source
5. snapshot
6. createdAt

## 9. 日历视图数据组织

前端不要直接拿原始 Plan 做复杂计算，建议做一层 Calendar ViewModel。

按天聚合后的结构建议：

1. dateKey
2. totalCount
3. eventCount
4. reminderCount
5. completedCount
6. overdueCount
7. items

这样月视图渲染会更轻，切换月份也更稳定。

## 10. 同步与来源设计

Plan 必须区分“数据主来源”。

建议定义：

1. App created
   由应用创建，再同步到 iOS
2. Imported from iOS
   由 iOS 导入到应用

MVP 阶段建议只做第一种，也就是：

- App 是主源
- iOS 是同步目标

不要一开始做全量双向同步，否则会直接进入冲突地狱。

## 11. iOS 联动映射规则

### 11.1 提醒事项

适合同步：

1. 交房租
2. 买东西
3. 明天提醒我联系某人
4. 截止类任务

映射字段建议：

1. title -> reminder title
2. notes -> reminder notes
3. dueAt -> due date
4. reminderAt -> alarm
5. status completed -> completed
6. externalId -> iOS reminder identifier

### 11.2 日历

适合同步：

1. 开会
2. 看医生
3. 约饭
4. 出行安排

映射字段建议：

1. title -> event title
2. notes -> event notes
3. startAt -> start date
4. endAt -> end date
5. isAllDay -> all day
6. reminderAt -> alarm
7. externalId -> iOS event identifier

## 12. 交互细节建议

### 12.1 月历点击行为

1. 点日期空白区域
   打开当天详情抽屉
2. 长按日期
   快速新建计划，默认日期填当天
3. 点计划点位
   打开详情卡片

### 12.2 计划详情页

详情页建议展示：

1. 标题
2. 时间
3. 提醒
4. 同步目标
5. 同步状态
6. 创建来源
7. 编辑按钮
8. 完成按钮
9. 删除按钮

### 12.3 冲突展示

发生同步冲突时，不要静默覆盖。

至少展示：

1. 应用内版本
2. iOS 版本
3. 最近修改时间
4. 选择保留哪一份

## 13. 与现有页面的结合方式

结合现有结构，建议这样接入：

1. 在 ViewMode 新增 PLAN
2. 在底部导航加“计划”入口
3. 新建 components/PlanCenter.tsx
4. PlanCenter 内部负责 今日 / 日历 / 清单 三个子视图
5. Logger 的一句话输入支持识别并转入 Plan 创建确认流程

建议不要把日历直接塞进 History 页面。

History 仍然负责日志回顾，PlanCenter 负责计划管理。

## 14. MVP 范围

第一阶段建议只做这些：

1. Plan 数据表和基础 CRUD
2. 一句话识别计划并结构化解析
3. 今日页
4. 月视图日历
5. 清单页
6. App -> iOS 单向同步
7. 同步状态展示

先不做：

1. 双向自动监听同步
2. 重复规则复杂编辑器
3. 多日程拖拽排布
4. 与 Goal 的自动联动

## 15. 视觉建议

你的应用当前是卡片化、轻拟物一点的移动端壳子，因此日历不要做成企业后台风格。

建议视觉方向：

1. 月视图卡片化
2. 日期格子圆角明显
3. 使用浅琥珀 + 深蓝灰作为主色
4. 当天日期明显高亮
5. 计划用小色条，不要一格里塞满文字
6. 详情用底部抽屉，不要频繁整页跳转

## 16. 研发顺序建议

推荐实现顺序：

1. 补齐 Plan 类型和接口
2. 做 PlanCenter 页面骨架
3. 做今日页和清单页
4. 做月历
5. 接一句话 AI 解析
6. 接 iOS 同步

这样能先把产品工作台搭起来，再去接原生桥，节奏更稳。

## 17. 最终结论

你这个需求不仅要做 Plan，还必须做 App 内日历。

正确结构应该是：

1. Goal 继续做连续习惯
2. Plan 负责行程和提醒
3. App 内日历作为 Plan 的主视图
4. iOS 提醒事项和日历作为同步目标，而不是主界面

如果后续继续开发，建议下一步直接进入接口和数据结构设计，而不是先写 UI。
